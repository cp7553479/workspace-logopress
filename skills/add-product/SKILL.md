# add-product

在 `商品表` 中新增或编辑商品记录。先收集信息、整理确认表，再写入飞书；不要未经确认直接落表。

**注意：**
以下 base-token、table-id、view-id、field-id 均为示例值，请根据实际情况替换成实际值，先读取表结构，再思考下一步。
如果未声明base token或无法准确定位表格请直接询问用户。
请回复用户，充分与用户沟通。

## 执行前先做

- 先用 `lark-cli base` 读取目标表的真实字段结构，确认字段名、字段类型、只读字段和多选字段选项。
- 涉及多维表格写入时，遵守 `lark-cli base` 与 `lark-cli shared` 的认证、安全和字段写入规则。
- 计算大货订货数或 P1 时，必须通过 bash 执行 `scripts/calc_product_metrics.py`，不要心算或口算。

## 核心工作流

### 1. 先判断是新增还是编辑

- 若用户说“新增商品 / 新增商品线索 / 收录新商品”，按新增处理。
- 若用户说“修改 / 编辑 / 补充商品”，按编辑处理。
- 若用户未明确说明，但提供了现有产品编号并要求修改信息，按编辑处理。

### 2. 读取表结构，再决定写法

先确认当前的base token的值为多少：
- base-token: `<your base token>`
然后确认`商品表`的Table ID为多少：
```bash
lark-cli base +table-list \
  --base-token  <your base token>
```

先执行以下命令读取真实字段结构：

```bash
lark-cli base +field-list \
  --base-token <your base token> \
  --table-id <your table id> \
  --offset 0 \
  --limit 200
```

只基于真实字段结构执行，不要猜字段名；不要把 formula、lookup、attachment 等特殊字段当作普通文本字段写入。

### 3. 收集并检查用户信息

优先确认这些字段：

| 字段 | 要求 |
|---|---|
| 产品编号 | 先找当前最大编号，再向用户建议下一个编号是多少，征求用户确认 |
| 参考图片 | 尽量提供；可辅助推测非关键字段，但不能据此推断价格、箱规、重量 |
| 来源(1688) | 1688 产品链接；缺失则留空 |
| 产品名称 | 必填，可通过产品图片判断 |
| 完整描述 | 必填，尽量完整 |
| 产品材质 | 可由图片或标题辅助推测，但要先列入确认表 |
| 产品颜色 | 多颜色时用英文逗号分隔 |
| 印刷方式 | 若未指定，给出建议，常见为 `Full Color,Silkscreen,Unimprinted`；也可能是 `Laser Engraved,Heat Transfer,UV printing` |
| 1688价(元) | 必须用户提供，不可猜；缺失则留空 |
| 箱子尺寸 长(cm) | 必须用户提供，不可猜；缺失则留空 |
| 箱子尺寸 宽(cm) | 必须用户提供，不可猜；缺失则留空 |
| 箱子尺寸 高(cm) | 必须用户提供，不可猜；缺失则留空 |
| 重量(箱子+产品)公斤 | 必须用户提供，不可猜；缺失则留空 |
| 件数/箱 | 必须用户提供，不可猜；缺失则留空 |
| 产品尺寸 长(cm) | 必须用户提供，不可猜；缺失则留空 |
| 产品尺寸 宽(cm) | 必须用户提供，不可猜；缺失则留空 |
| 产品尺寸 高(cm) | 必须用户提供，不可猜；缺失则留空 |
| 产品重量(kg) | 必须用户提供，不可猜；缺失则留空 |
| Q1~Q10 | 阶梯订货量；Q1 常作为 MOQ，但不要自动视为大货订货数 |
| P1~P10 | 阶梯价；P1 可在条件足够时计算，P2~P10 默认由用户提供，除非后续补充明确公式 |
| D1~D10 | 默认 `C / R`，除非用户另有说明 |
| 产品类型 / 分类 / 简要说明 / 产品关键词 / 主题 | 可辅助推测，但先列入确认表 |

## 不可猜测规则

以下字段严禁直接猜：
- 1688价(元)
- 箱子尺寸 长(cm)
- 箱子尺寸 宽(cm)
- 箱子尺寸 高(cm)
- 重量(箱子+产品)公斤
- 件数/箱
- 产品尺寸 长(cm)
- 产品尺寸 宽(cm)
- 产品尺寸 高(cm)
- 产品重量(kg)

缺失时：
- 表格中留空
- 在回复里提醒用户这些值仍缺失
- 不要在字段值中写“待确认”“推测值”等说明文字
- 友好地引导式向用户提问，争取让用户补充完整信息，调整你的回答的排版，关键信息可加粗

## 推测规则

可以基于参考图片、产品名称、完整描述推测：
- 产品材质
- 产品颜色
- 印刷方式建议
- 产品类型
- 分类
- 简要说明
- 产品关键词
- 主题

但必须遵守：
- 先把推测结果放进给用户确认的表格里
- 不要把“推测”之类的标志性的文字写进表格里
- 如需解释理由，优先写在回复里；只有在用户明确接受时，才把必要说明写进 `备注`

## 确认表要求

在真正写入飞书前，必须先输出一张表格给用户确认。至少包含：
- 字段
- 拟写入值

表格里不要出现：
- `此处留空，待确认`
- `此为推测值`
- 任何类似标记性说明文字

缺失字段直接留空；解释写在表格外的正文里。

## 新增流程

### 1. 若产品编号缺失

不要直接询问用户产品编号是多少，而是寻找当前表格产品编号的最大值往后推一位，询问用户是否可以用这个产品编号。用 `lark-cli base +data-query` 过滤 `产品编号` 中的记录，并按 `产品编号` 倒序取第 1 条，再建议下一个编号。

示例代码：

```bash
lark-cli base +data-query \
  --base-token <your base token> \
  --dsl '{
    "datasource":{"type":"table","table":{"tableId":"<your table id>"}},
    "dimensions":[{"field_name":"产品编号","alias":"product_number"}],
    "filters":{
      "type":1,
      "conjunction":"and",
      "conditions":[
        {"field_name":"产品编号","operator":"contains","value":["LP"]}
      ]
    },
    "sort":[{"field_name":"product_number","order":"desc"}],
    "pagination":{"limit":1},
    "shaper":{"format":"flat"}
  }'
```

然后：
1. 读取返回的第 1 条 `产品编号`。
2. 解析其中的数字部分。
3. 生成下一个编号。
4. 问用户是否可以使用该编号。
5. 用户确认后再继续。

编号比较规则：
- 该排序方法依赖编号格式尽量统一；若位数不统一或格式混乱，再退回到全量读取后提取数字比较。
- 若遇到不规范编号，不把它当作最大值依据。

### 2. 先整理确认表

把用户提供值、可推测值和缺失字段整理成表格，让用户确认。

### 3. 条件足够时计算数值

#### 大货订货数

当以下字段完整时才计算：
- 箱子尺寸 长(cm)
- 箱子尺寸 宽(cm)
- 箱子尺寸 高(cm)
- 重量(箱子+产品)公斤
- 件数/箱

公式：

`ROUNDUP(15 ÷ (Max(箱子尺寸长*箱子尺寸宽*箱子尺寸高 ÷ 6000, 重量(箱子+产品)公斤) ÷ 件数/箱), 0)`

#### P1

当以下字段完整时才计算：
- 箱子尺寸 长(cm)
- 箱子尺寸 宽(cm)
- 箱子尺寸 高(cm)
- 重量(箱子+产品)公斤
- 件数/箱
- 1688价(元)

公式：

`ROUNDUP(((Max(箱子尺寸长*箱子尺寸宽*箱子尺寸高 ÷ 6000, 重量(箱子+产品)公斤) ÷ 件数/箱 * 25 + 1688价(元)) ÷ (1 - 0.3) ÷ 7), 2)`

### 4. 用脚本计算

通过 bash 执行：

```bash
scripts/calc_product_metrics.py \
  --carton-length <cm> \
  --carton-width <cm> \
  --carton-height <cm> \
  --carton-weight <kg> \
  --pcs-per-carton <qty> \
  --price-1688 <price> \
  --exchange-rate 7
```

返回 JSON 字段：
- `big_cargo_qty`
- `p1`（美元价）
- `volume_weight_6000`
- `chargeable_weight_per_carton`
- `chargeable_weight_per_piece`

若计算条件不足，不要硬算。

### 5. 用户确认后再写入

新增记录时：
- 使用 `record-upsert` 的创建模式
- 只写可写字段
- 附件字段通过附件上传链路处理

## 编辑流程

### 1. 用产品编号定位记录

- 先按 `产品编号` 搜索记录。
- 若命中 0 条，告诉用户没找到。
- 若命中多条，先把候选记录列出来让用户确认。
- 若命中 1 条，再整理“修改后确认表”。

### 2. 给用户看修改后表格

编辑时优先用“字段 / 原值 / 新值”表格展示；至少要让用户看到最终拟写入值。

### 3. 用户确认后更新

- 使用 `record-upsert --record-id <record_id>` 更新。
- 仍然只写可写字段。

## 附件处理

- 不要把附件字段当成普通 JSON 文本写入。
- 若用户提供图片文件或链接，按附件上传链路处理。
- 若当前链路暂时不能上传附件，先完成文本字段确认，并明确告知附件待补。

## 默认建议

### 印刷方式默认建议

若用户未明确提供，按材质/常见场景给建议：
- 常规塑料/金属/木制表面：`Full Color,Silkscreen,Unimprinted`
- 金属、木材、竹材：可补充 `Laser Engraved`
- 布料类：可补充 `Heat Transfer`
- 平整硬质表面：可考虑 `UV printing`

建议写在确认表的拟写入值中，但仍需用户确认。

### 折扣代码默认值

- D1~D10 默认填 `C / R`
- 若用户另有折扣体系，以用户提供为准

### 阶梯价补全示例

- `Q1` 默认取“大货订货数”和“件数/箱”中的较高值。
- `Q2~Q10` 可按阶梯提升，例如：`50, 100, 200, 500, 1000, 2000, 5000, 10000, 20000, 50000`。
- 已知 `P1` 时，可在后续档位做平滑递减。
- 每一档价格执行 `ROUNDUP(value, 2)`，向上保留两位小数。
- 这是示例，不要未经用户确认直接套用。

## 写入约束

- 不要写只读字段，尤其是 formula 和 lookup 字段。
- 不要把解释性文字写进业务字段。
- 不要在用户未确认前直接新增或修改。
- 不要假设 Q1 一定等于大货订货数；只能说“一般等同”，最终仍以业务确认为准。
- 不要因为用户给了图片就推断箱规、重量、1688价。

回复时优先给出确认表、缺失字段与待确认事项。若信息不完整，直接指出缺什么，不要模糊带过。
