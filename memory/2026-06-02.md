
## ASI batch continuation lapse
- Trigger: During the 51-product ASI image batch, LP0556 completed, was verified, images were sent, and progress was updated, but the next product was not immediately started.
- Evidence: User asked why no active runner existed; subagent list showed none after LP0556; progress still had pending/queued products.
- Cause: I treated the product completion/report as a turn endpoint instead of enforcing the batch invariant: if the batch is unfinished, always start or schedule the next runner before stopping.
- Lesson: For long batch workflows, after every product completion event run this checklist: verify outputs, upload/verify, send images, update progress, then immediately start the next eligible product or record a concrete blocker.
- Guardrail: Added `HEARTBEAT.md` task `asi-batch-runner-guard` to catch unfinished ASI batches with no active runner.
